# DataWhale 深度设计分析

> 基于 OmniData 设计文档 + OpenCode / Oh My OpenCode / Pi-Mono 的对比研究

---

## 一、先回答核心问题：什么是 "AI Native" 的数据 Agent？

### 1.1 当前 OmniData 的本质：AI-Wrapped BI

OmniData 的架构本质上是在传统 BI 栈上叠加了一层 AI：

```
传统 BI 层 (保留)          AI 层 (叠加)
─────────────────────    ─────────────────
• 连接器体系 (JDBC...)   • NL2SQL (Agent 翻译)
• 查询引擎 (Trino)       • 对话式交互
• ETL 管道               • 多 Agent 角色分工
• 可视化引擎             • 智能图表推荐
• 权限/审计              • 自然语言报告生成
```

这不是 AI Native —— 这是 **AI Wrapper**。AI 在这里是一个"翻译层"：把自然语言翻译成 SQL，然后丢回传统管道执行。核心数据处理的逻辑、瓶颈、和范式没有改变。

### 1.2 AI Native 的定义

AI Native 意味着 **AI 不是附加功能，而是系统的第一性原理**。类比：

| 层面 | AI-Wrapped (OmniData) | AI-Native (DataWhale 目标) |
|------|----------------------|---------------------------|
| **数据建模** | 人工定义 Schema，AI 辅助查询 | AI 自主理解数据语义，动态构建知识图谱 |
| **查询执行** | NL → SQL → 传统引擎 | NL → 多策略探索（SQL + 语义搜索 + 向量检索 + 代码执行） |
| **Agent 角色** | 固定角色（分析师/工程师/可视化） | 按需实例化，动态能力组合 |
| **知识积累** | 会话历史存储 | 跨会话持续学习，知识蒸馏 |
| **扩展方式** | 插件注册 | Agent 自扩展（Agent 写 Agent） |
| **错误处理** | 重试 + 人工介入 | 自我修复 + 策略回退 |
| **人机关系** | 对话驱动（人主导每一步） | 委托驱动（人表达意图，Agent 自治执行） |

## 二、三个参考项目的核心启示

### 2.1 OpenCode —— "少即是多"的 Agent 设计

**核心发现：只有 2 个 Agent（build + plan）就覆盖了全部场景。**

```
build  Agent: 全能力，做开发
plan   Agent: 只读，做分析/规划
general SubAgent: 复杂搜索和多步任务（内部调用）
```

**启示：**
- Agent 不需要按领域硬编码角色（分析师、工程师、可视化...），这种角色划分是传统组织架构的投影，不是 AI 最优解
- 应该按 **能力边界** 而不是 **职能标签** 来定义 Agent
- 并行工具调用是第一优先级

### 2.2 Pi-Mono —— "自扩展"架构的终极形态

**核心发现：Pi 不内置 sub agent、plan mode，但用户可以用 TypeScript 扩展一切，甚至可以用 Pi 来扩展 Pi。**

```
pi-ai         → 统一多提供商 LLM API
pi-agent-core → Agent 运行时（工具调用 + 状态管理 + 事件流）
pi-coding-agent → CLI + Extensions + Skills + Prompt Templates + Themes
```

**关键设计：**
- **Extensions**：TypeScript 文件，加载后增强 Agent 能力
- **Skills**：可复用的提示词/知识模块
- **Prompt Templates**：可组合的系统提示
- **四种模式**：交互式 / 非交互式 / RPC / SDK 嵌入
- **开源 session 数据共享**：真实工作流 data → 改进模型

**启示：**
- Agent 的核心应该极度精简，能力通过"外挂"而非"内置"获得
- 自扩展（Agent 写 Extension）是终极灵活性
- 开源 session 数据是最被低估的护城河

### 2.3 Oh My OpenCode —— "委托式自治"的激进实验

**核心发现：Ultrawork 宣言重新定义了人机关系。**

```
人的干预 = 系统失败信号
目标：代码无法区分是人写的还是 Agent 写的
```

**关键架构：**

| 组件 | 职责 |
|------|------|
| **Sisyphus** | 主编排器 |
| **Prometheus** | 访谈式需求澄清（Planner） |
| **Metis** | 计划顾问，在歧义变成 Bug 前捕获 |
| **Momus** | 计划完整性验证 |
| **Orchestrator** | 无微管理的协调 |
| **Todo Continuation** | 强制完成，防止 AI "我已做完"的谎言 |
| **Category System** | 按任务领域路由到最优模型 |
| **Background Agents** | 并行调研，不阻塞用户 |
| **Wisdom Accumulation** | 跨任务学习，不重复犯错 |

**启示：**
- "人只表达意图，Agent 自治执行" 是更高维度的人机范式
- 多模型动态路由（便宜模型做简单活，强模型做复杂推理）是成本和质量的最优解
- 计划验证（Momus）和完成强制（Todo Continuation）解决了 AI Agent 最大的可靠性问题
- 知识积累是 Agent 进化的核心引擎

## 三、OmniData 的核心局限（批判性分析）

### 3.1 架构层面的问题

**局限 1：传统分层架构与 AI 工作流的冲突**

OmniData 的分层设计（表现层 → 网关 → 应用层 → 领域层 → 基础设施）是传统微服务架构的投影。但 AI Agent 的工作流是 **非线性的、探索性的、跨层跳跃的**：

```
传统分层调用：A → B → C → D（严格自顶向下）
Agent 实际行为：A → D → B → A → C（跳跃、复查、自我修正）
```

**局限 2：Agent 角色固化的认知偏差**

Analyst、Engineer、Statistician、Visualizer 这四个角色是最典型的"把人类组织架构映射到 AI"的错误。在真实的数据分析中：
- 一个优秀的分析师同时在做工程、统计、可视化
- 角色边界模糊，强制分离只会增加通信开销
- AI 的优势恰恰在于跨界整合，而非专业分工

**局限 3：数据处理引擎的 AI 盲区**

查询引擎仍然是传统模式：SQL 解析 → 优化 → 执行。但 AI 时代的数据处理应该是：
- **语义级别的数据理解**，而非 Schema 级别的
- **多策略探索**：SQL、语义搜索、代码执行、向量检索并行
- **自动数据发现**：Agent 主动探索数据湖，而非被动等待用户指定数据源

**局限 4：会话级的状态管理**

OmniData 以 Session 为状态边界，会话之间不共享知识。这导致：
- 每次新分析从零开始
- 无法积累"这个数据库的 orders 表实际含义是 XX"这类隐性知识
- 用户重复澄清相同的问题

### 3.2 范式层面的问题

**局限 5：对话驱动 vs 委托驱动**

OmniData 本质是"对话驱动"——用户问一句，Agent 回一句。这在简单场景下够用，但在复杂分析中变成了"人在循环中"的瓶颈。

Oh My OpenCode 的 Ultrawork 哲学提出了更高维度：**人表达意图 → Agent 自治执行 → 人验收结果**。

**局限 6：缺乏自扩展能力**

OmniData 的扩展方式是传统的插件注册。但最先进的范式是 **自扩展**：用 Agent 来扩展 Agent 的能力。如 Pi 的设计——你可以让 Pi 帮你写一个 Extension 来扩展 Pi。

## 四、DataWhale 的核心设计理念

基于以上分析，DataWhale 的设计应该围绕以下核心原则：

### 原则 1：意图驱动，而非对话驱动

```
OmniData:  用户 → 提问 → Agent 回答 → 用户追问 → ...
DataWhale: 用户 → 表达意图 → Agent 自治探索 → Agent 汇总结果 → 用户验收
```

用户说"帮我分析一下这个季度的销售异常"，Agent 应该：
1. 自动探索可用的数据源
2. 理解数据语义（哪个表是销售？时间维度是什么？）
3. 设计多角度分析策略
4. 并行执行（趋势分析 + 异常检测 + 归因分析 + 竞品对比）
5. 自我验证结果一致性
6. 汇总为结构化报告
7. 只在真正需要决策时才询问用户

### 原则 2：能力边界 > 角色标签

不预设 Analyst / Engineer / Statistician 等角色。改为：

```
按能力维度定义 Agent：
┌─────────────────────────────────────────────┐
│  DataWhale Agent 能力矩阵                     │
│                                              │
│  • 数据发现 (Data Discovery)                 │
│  • 语义理解 (Semantic Understanding)         │
│  • 查询生成 (Query Generation)               │
│  • 统计推理 (Statistical Reasoning)           │
│  • 可视化设计 (Visual Design)                │
│  • 故事叙述 (Narrative Building)             │
│  • 代码执行 (Code Execution)                 │
│  • 知识检索 (Knowledge Retrieval)            │
│                                              │
│  每个任务实例按需组合能力，而非绑定角色       │
└─────────────────────────────────────────────┘
```

### 原则 3：自扩展架构

```
DataWhale Core (极简内核)
├── Agent Runtime (pi-agent-core 风格)
├── Tool Registry
├── Event Bus
└── Knowledge Graph

DataWhale Extensions (自扩展层)
├── 内置 Extensions (Powered by DataWhale itself)
├── 社区 Extensions (npm/git 分发)
└── 用户 Extensions (Agent 辅助生成)
```

用户可以用 DataWhale 来扩展 DataWhale——这是 AI Native 的标志性特征。

### 原则 4：多模型动态路由

借鉴 Oh My OpenCode 的 Category System：

```
任务分类 → 模型选择 → 成本优化

简单查询 (SELECT * FROM...)  → 便宜模型 (Haiku/Flash)
复杂分析 (多维归因)           → 强推理模型 (Sonnet/Opus)
代码生成 (Python/SQL)         → 代码专长模型
可视化设计                    → 多模态模型
知识检索                      → Embedding + 向量搜索
```

### 原则 5：跨会话知识积累

```
Session 1: 用户分析"华东区销售" 
  → 学习: orders.region='east' 对应华东区
  → 学习: 销售数据存在 2 天延迟
  → 存入 Knowledge Graph

Session 2: 用户分析"全国销售趋势"
  → 自动应用: region 过滤逻辑
  → 自动提醒: "注意，销售数据有 2 天延迟"
  → 无需用户重复澄清
```

### 原则 6：开源 Session 数据

借鉴 Pi 的开源 session 共享模式。数据分析 session 数据（脱敏后）是改进数据 Agent 最宝贵的训练数据——比任何合成 benchmark 都更有价值。

## 六、关键设计决策深度分析

### 6.1 为什么取消固定角色 Agent？

OmniData 的四角色模型（Analyst、Engineer、Statistician、Visualizer）看起来合理，但存在根本性问题：

1. **角色粒度粗暴**：真实数据分析中，同一人可能同时做"工程师的 SQL 调优"和"统计学家的假设检验"，强制分离增加编排复杂度
2. **边界模糊导致委托失败**：主 Agent 需要判断"这个任务该给 Engineer 还是 Statistician？"——这个判断本身就是高认知负载的
3. **通信开销**：Agent 之间的上下文传递是损耗点，减少 Agent 数量 = 减少信息熵增

**替代方案：能力组合 + 动态实例化**

```
不预设 Agent 角色，而是：
1. 定义原子能力 (Tool) 池
2. 每个具体任务，由 Orchestrator 根据任务特征动态组合能力
3. 任务完成后，能力实例即释放
```

### 6.2 为什么需要知识图谱而不是纯向量存储？

OmniData 的"会话历史"本质上是非结构化的。但数据分析领域天然是结构化的：

```
数据源 → 表 → 列 → 值域 → 关系

向量搜索适合"语义相似"（"销售和营收是一回事吗？"）
知识图谱适合"精确关系"（orders.region = 'east' 就是华东区）
```

两者结合：向量做模糊匹配，图谱做精确推理。

### 6.3 为什么强调"自扩展"？

这不仅是一个架构选择，更是一个哲学立场：

- 如果 DataWhale 的 Extension 系统足够好，用户可以用 DataWhale 来写 Extension 扩展 DataWhale
- 这创造了一个 **正反馈循环**：更好的 Extension → 更强的 Agent → 写更好的 Extension
- 这解决了"核心团队永远无法满足所有长尾需求"的问题

### 6.4 为什么需要多模型动态路由？

单模型策略（OmniData 默认 Sonnet）存在的问题：
- 简单 SELECT * FROM table 用 Sonnet 是严重浪费
- 不同模型在不同任务上的表现天差地别
- 成本是 AI Agent 大规模使用的核心瓶颈

```
任务复杂度 → 模型选择 → 预估成本

Level 1: 简单元数据查询 → Haiku/Flash (极低成本)
Level 2: 单表分析       → Sonnet/Haiku 混合
Level 3: 多维归因分析   → Sonnet (高推理能力)
Level 4: 策略建议       → Opus/GPT-4 (最强推理)
Level 5: 可视化设计     → 多模态模型
```

## 七、技术栈建议

| 层面 | 选型 | 理由 |
|------|------|------|
| **语言** | TypeScript (Bun runtime) | 与 OpenCode/Pi 生态一致，前后端统一 |
| **AI SDK** | Vercel AI SDK + 自研 Provider 抽象 | 多模型统一调用 |
| **Agent Runtime** | 借鉴 pi-agent-core 设计 | 事件驱动、streaming、工具调用 |
| **知识图谱** | 嵌入式 (如 SQLite + 向量扩展) | 轻量、无外部依赖 |
| **向量存储** | LanceDB 或 SQLite-vss | 嵌入式、零运维 |
| **查询执行** | DuckDB (嵌入式) + 连接器体系 | 分析型工作负载最优 |
| **Python 沙箱** | Pyodide 或 WASM | 安全隔离执行数据分析代码 |
| **前端** | React + TUI (终端) + API | 多界面策略 |
| **会话存储** | SQLite + 可选 PostgreSQL | 单机 → 团队无缝升级 |
| **扩展系统** | ESM 动态 import + npm/git | Pi 的扩展模式验证过 |

## 八、优先级与演进路径

### Phase 1: Core Agent + Simple Data（2-3 月）

- 极简 Agent Runtime（借鉴 pi-agent-core）
- 能力池设计（数据发现、查询生成、代码执行）
- DuckDB 嵌入式查询
- 基础 Extension 系统
- 单模型（先跑通闭环）

### Phase 2: Knowledge + Multi-Model（3-4 月）

- 知识图谱 + 向量存储
- 多模型动态路由
- 跨会话学习
- 意图解析 + 计划验证
- TUI + CLI 交互

### Phase 3: Autonomy + Ecosystem（4-6 月）

- 委托式自治执行
- 自扩展（Agent 写 Extension）
- Session 数据开源
- 社区市场
- Web UI + API

## 九、风险与挑战

1. **"意图驱动"的实现难度极高**：让 Agent 准确理解用户真实意图，需要比对话驱动强得多的推理能力
2. **知识图谱的冷启动**：没有数据时，图谱是空的，需要设计优雅的渐进式学习机制
3. **自扩展的安全边界**：Agent 写的代码执行权限需要精细控制
4. **多模型路由的准确性**：任务分类错误会导致成本浪费或质量下降
5. **与现有数据生态的兼容**：企业已有 Tableau/BI 系统，DataWhale 如何共存而非替代

## 十、总结

**DataWhale 不是一个"更好的 OmniData"，而是一个范式转移：**
