## 一、可视化方案评估

### 终端可视化库对比

| 库 | 语言 | 图表类型 | 适合场景 |
|----|------|---------|---------|
| **Rich** | Python | 表格、语法高亮、Markdown | Python 生态标准（50k+ ⭐） |
| **Plotext** | Python | 折线/柱状/散点/直方图/热力图 | matplotlib 风格终端绘图 |
| **asciichart** | JS/Python | 折线图 | 零依赖、轻量 |
| **blessed-contrib** | Node.js | 仪表板全套 | 全终端仪表板 |
| **termichart** | Rust+多语言 | 8 种图表 | 高性能跨平台 |

### DataWhale 推荐方案

**双层策略**：

```
Layer 1: 即时 ASCII 输出（纯 TypeScript，零依赖）
  ├── 查询结果 → 自动格式化 ASCII 表格
  ├── 数字序列 → 自动生成 sparkline ▁▂▅▇█▆▄▃
  └── 简单对比 → Unicode 柱状图 ████████░

Layer 2: E2B 图像生成（已有沙箱，直接复用）
  ├── Agent 写 matplotlib → E2B 执行 → 导出到 ~/.datawhale/plots/
  └── 用户 open ~/.datawhale/plots/ 查看
```

**为什么不用 Rich/Plotext**：它们是 Python 库，需要通过 E2B 沙箱执行，而纯 TypeScript 的 ASCII 图表可以零延迟直接在终端渲染。

**现实对标**：Claude Code 和大多数 AI coding agent 都走这条路 —— Markdown 格式化输出 + matplotlib 图像兜底，不在终端做复杂图表。

### 实现清单

| 组件 | 方式 | 估时 |
|------|------|------|
| ASCII 表格渲染 | TypeScript 手写（~50行） | 0.2天 |
| sparkline 生成 | ▁▂▃▄▅▆▇█ unicode | 0.1天 |
| 柱状图 | ████░ unicode bar | 0.1天 |
| E2B matplotlib 联动 | 复用现有沙箱 | 已有 |
| 总计 | | **0.5天** |

## 二、知识积累方案评估

### 向量数据库对比

| 特性 | LanceDB | Chroma | Pinecone |
|------|---------|--------|----------|
| 部署 | 嵌入式、本地 | 嵌入式 | 云托管 |
| 延迟 | 极低（进程内） | 低 | ~5-10ms |
| 成本 | 免费开源 | 免费开源 | 按量付费 |
| 多模态 | ✅ 原生 | ❌ | 有限 |
| 适合 | **AI Agent 本地记忆** | 原型验证 | 生产高规模 |

### 关键发现

1. **Mem0** — 2025年最受关注的 Agent 记忆框架，支持记忆自动提取/去重/重要性评分，**70% 任务完成率提升**
2. **Letta (MemGPT)** — 操作系统风格的记忆管理（工作记忆/长期记忆/归档），**26% 更高准确率**
3. **GraphRAG** (微软) — 知识图谱增强 RAG，比传统 RAG 在需要全局理解的任务上显著更好
4. **行业共识**：Agent 记忆应该分层 —— 工作记忆（上下文窗口）、情景记忆（向量库）、语义记忆（知识图谱）

### DataWhale 推荐方案

**三步走，渐进式**：

```
Step 1: 轻量知识库（0.5天）
  ├── 会话结束时 LLM 提取关键发现
  ├── 存入 SQLite 表（无外部依赖）
  └── 下次会话自动注入 System Prompt

Step 2: 向量语义搜索（1天）
  ├── 安装 LanceDB（嵌入式，Bun 兼容待验证）
  ├── DeepSeek Embedding API 向量化
  └── 语义匹配历史知识 → 自动注入

Step 3: 轻量知识图谱（1天）
  ├── SQLite 关系表存储 schema 语义
  ├── "华东区" → region='East' 映射
  └── 自动发现表间 JOIN 关系
```

**为什么不是直接用 Mem0/Letta**：它们是通用记忆框架，对 DataWhale 的数据分析场景过重。DataWhale 的记忆需求更聚焦——记住数据语义和查询模式，而非通用对话记忆。

## 三、模型路由方案评估

### 路由平台对比

| 平台 | 方式 | 成本节省 | 适合 |
|------|------|---------|------|
| **LiteLLM** | 自建代理 | 70-90% | 需要完全控制 |
| **OpenRouter** | API 聚合 | 70-90%（自动） | 快速接入 400+ 模型 |
| **NotDiamond** | 学习路由 | 最优 | 高流量训练 |
| **Martian** | 模型映射 | 20-97% | 企业级 |

### 关键发现

1. **约 80% 的请求是简单查询**（元数据、COUNT、简单 SELECT）—— 可以用便宜模型
2. **加权平均可降低 86% 成本**（简单查用 Flash，复杂分析用 V3/Sonnet）
3. **不需要昂贵的路由服务**——简单分类器 + 3 层模型栈即可获得大部分收益
4. **Prompt Caching 是免费午餐**——Anthropic 最高省 90% 输入成本，应在所有 provider 上启用

### DataWhale 推荐方案

**启发式路由（最简单有效）**：

```
用户请求 → 复杂度评估 → 模型选择

Level 1 (80%): 简单查询
  特征：无 JOIN、无聚合、元数据查询
  → DeepSeek-V3 或 Flash（极低成本）

Level 2 (15%): 中等分析
  特征：含 GROUP BY、2表 JOIN、统计分析
  → DeepSeek-V3（主力模型）

Level 3 (5%): 复杂推理
  特征：多维归因、策略建议、需要 Python 沙箱
  → DeepSeek-R1 或 Sonnet（强推理）
```

实现：纯规则判断（检查 SQL 特征 + 工具调用次数），**估时 0.5 天**。

未来可升级为 LLM 自判断（让 Agent 自己决定用哪个模型）。
